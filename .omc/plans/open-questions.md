# claudehome — Open Questions

## claudehome-folder-tree-v1 — 2026-05-08
- [ ] OQ-1: When the folder-vs-project heuristic classifies a `.git`-only directory as a folder, should we add a special-case override (".git as the only child → project")? — Why it matters: would silently break drilling into solo-git repos. Spec says "revisit if it produces surprises"; defer to first user report.
- [x] OQ-2 (RESOLVED 2026-05-08, iter 2): `head -z -n 2000` portability across macOS BSD `head`. Resolution: BSD `head` rejects `-z` outright (`head: invalid option -- z`). Plan now uses `awk -v RS='\0' -v ORS='\0' 'NR<=2000'` everywhere, portable across macOS bash 3.2 / GNU / BSD. The bare `awk 'NR<=2000'` form is also wrong (default `RS='\n'` consumes the entire NUL-stream as one record), so the explicit `-v RS='\0' -v ORS='\0'` flags are mandatory. No longer blocks Phase 1.

## claudehome-5type-v1 — 2026-05-08
- [ ] OQ-3: Should the picker offer to "promote" a Folder to Project on user demand (i.e., touch CLAUDE.md inside it)? Closes the raw-`mv`-into-Suite workflow without forcing the user to drop to a shell. — Why it matters: a user who moves a Project into a Suite via raw `mv` may want a one-click way to keep it as Project (not let it become a Member transitively); deferred to first user request.
- [ ] OQ-4 (added iter-2 from Architect P9 nice-to-have): Should v1.1 move H/M synthesis from the parser to the server-side emitter for wire-format-vs-parser symmetry? The current 5-type plan's parser regex `^[RFSPHM]$` accepts H and M codes the server never emits (server emits P for has-CLAUDE.md; parser materializes H/M via ancestor walk). — Why it matters: clean wire-format invariants are easier to audit; future contributors may misread H in the parser regex as dead code. Defensible as-is per Architect — defer until measured bugs or 500ms-worst-case parser walks become real complaints.
- [ ] OQ-5 (added iter-2 from Architect P10 nice-to-have): Should v1.1 switch Multi-Hub Suites from warn+skip to alphabetical-first-primary + warn? Current plan LOCKS warn+skip per spec line 79-80 + Non-Goals line 151. — Why it matters: this is the only major ergonomic concession in the spec; alphabetical-first is friendlier UX but locks in hidden behavior the user can't opt out of without renaming. Defer until first user reports a real multi-Hub state (vs. a misconfiguration).
- [x] OQ-1 (RESOLVED, was spec OQ-1): Suite-root drill rendering. Resolution: alphabetical-interleaved Folders+Suites, then active-then-idle Projects+Hubs+Members, then `[new...]`. Same ordering as every other drill level. No section headers, no separator rows. Documented in plan §3.3.
- [x] OQ-2 (RESOLVED, was spec OQ-2): Auto-suffix policy for Suite/Hub. Resolution (iter-1) auto-append `_suite`/`_hub` to user-typed name if missing; reject (with re-prompt) if user-typed name already ends with the suffix. **Resolution patched iter-2 (Critic P2):** rejection now catches BOTH suffix-form (`gene-mini_suite`) AND interior-substring form (`gene-mini_suite_v2`) via the bash glob `*_suite|*_suite_*` and pwsh `-match '_suite'`. Documented in plan §3.3 + §7 R10 + tested in §8.9; new ACs AC-5T19 and AC-5T20 cover the substring rejection.
